# ranking-service

Stage 2 ranking service for the HR resume recommendation system.

The service receives vacancy fields and Stage 1 retrieval candidates from the backend orchestrator, builds CatBoost-compatible pair features, scores candidates with the trained CatBoostRanker, and returns ranked CV ids with CatBoost scores.

The service does not calculate embeddings, does not run Stage 1 retrieval, and does not assemble frontend cards.

## Current status

Implemented:

- `GET /health`
- `GET /ready`
- `POST /rank`
- Pydantic request/response schemas
- vacancy normalization for Stage 2 feature building
- normalized CV feature loading by `cv_id_hash`
- pair feature construction
- CatBoostRanker artifact loading
- CatBoost scoring
- Dockerfile

Current ranking behavior:

- candidates are scored by CatBoostRanker;
- candidates are sorted by `model_score` descending;
- ties are resolved internally by `embedding_score` descending, then `embedding_rank` ascending;
- `result_limit` controls how many ranked candidates are returned, with default and max value `500`.

Important: `model_score` is a raw CatBoost ranking score. It is not a probability and it is not a percentage.

## Service role

The Stage 2 service is responsible for:

1. receiving candidates from Stage 1 retrieval;
2. loading normalized CV features by `cv_id_hash`;
3. normalizing incoming vacancy fields for Stage 2 feature building;
4. building CatBoost-compatible pair features;
5. scoring candidates with CatBoost;
6. returning ranked CV ids and CatBoost scores to the backend orchestrator.

The backend remains responsible for assembling the final frontend response.

## Runtime artifacts

The service expects runtime artifacts inside the container:

```text
/app/artifacts/models/ranker_manifest.json
/app/artifacts/models/ranker_feature_columns.json
/app/artifacts/models/<model_file>.cbm
/app/artifacts/cv/cv_normalized.parquet
```

The artifact paths are configured through environment variables:

```text
RANKING_RANKER_MANIFEST_PATH=/app/artifacts/models/ranker_manifest.json
RANKING_CV_STORE_PATH=/app/artifacts/cv/cv_normalized.parquet
```

Default values are already set in the service config, so these environment variables are optional when the artifacts are mounted to the default paths.

Artifact ownership in the full project:

```text
main/models/
  ranker_manifest.json
  ranker_feature_columns.json
  <model_file>.cbm

main/data/processed/v1/
  cv_normalized.parquet
```

`cv_normalized.parquet` is a local data artifact and is expected to be generated by the preprocessing step in the main repository.

## Run with Docker Compose

Recommended local run from the `main` repository root:

```bash
docker compose -f docker-compose.stage2.yml up --build ranking-service
```

Check readiness:

```bash
curl http://127.0.0.1:8002/ready
```

Stop service:

```bash
docker compose -f docker-compose.stage2.yml down
```

The compose file mounts:

```text
./models -> /app/artifacts/models
./data/processed/v1 -> /app/artifacts/cv
```

## Standalone Docker build

Build image from the `ranking-service` repository root:

```bash
docker build -t ranking-service .
```

Run with explicit artifact mounts from inside `main/ranking-service`:

```bash
docker run --rm -p 8002:8000 \
  -v ../models:/app/artifacts/models:ro \
  -v ../data/processed/v1:/app/artifacts/cv:ro \
  ranking-service
```

Open healthcheck:

## API

### `GET /health`

Returns process liveness.

This endpoint does not check whether artifacts are loaded.

Response:

```json
{
  "status": "ok"
}
```

### `GET /ready`

Returns service readiness.

If all artifacts are mounted and loaded, response is:

```json
{
  "status": "ready",
  "artifacts": {
    "ranker_manifest_path": "/app/artifacts/models/ranker_manifest.json",
    "ranker_manifest_exists": true,
    "cv_store_path": "/app/artifacts/cv/cv_normalized.parquet",
    "cv_store_exists": true,
    "model_path": "/app/artifacts/models/<model_file>.cbm",
    "model_exists": true,
    "feature_columns_path": "/app/artifacts/models/ranker_feature_columns.json",
    "feature_columns_exists": true
  }
}
```

If artifacts are missing or invalid, this endpoint returns `503`.

### `POST /rank`

Accepts vacancy fields and Stage 1 candidates.

Request shape:

```json
{
  "job_id": "job-1",
  "vacancy": {
    "vacancy_text": "Python backend developer",
    "profession": "developer",
    "group_profession": "IT",
    "business_category": "software",
    "sfera": "IT",
    "experience": "более 1 года",
    "schedule": "фиксированный",
    "employment_type": "полная занятость",
    "education_level": "не имеет значения"
  },
  "candidates": [
    {
      "cv_id_hash": "cv-1",
      "embedding_score": 0.91,
      "embedding_rank": 1
    },
    {
      "cv_id_hash": "cv-2",
      "embedding_score": 0.84,
      "embedding_rank": 2
    }
  ],
  "result_limit": 500
}
```

Response shape:

```json
{
  "job_id": "job-1",
  "ranked": [
    {
      "cv_id_hash": "cv-1",
      "rank": 1,
      "model_score": 1.2345
    },
    {
      "cv_id_hash": "cv-2",
      "rank": 2,
      "model_score": 0.1412
    }
  ]
}
```

The service returns up to `result_limit` candidates. For the standard Stage 1 top-500 input, use `result_limit=500`.

## Inference flow

```text
Stage2Request
  -> validate request
  -> load candidates from request
  -> load normalized CV features by cv_id_hash
  -> normalize vacancy fields
  -> build pair features
  -> create CatBoost Pool
  -> predict model_score
  -> sort candidates by model_score
  -> return Stage2Response
```

## Backend integration

The backend orchestrator calls this service through `STAGE2_URL`.

Example:

```text
STAGE2_URL=http://ranking-service:8000/rank
```

The backend sends a `Stage2Request`:

```text
job_id
vacancy
candidates
result_limit
```

Where `vacancy` contains vacancy fields that the ranking service normalizes for feature building:

```text
vacancy_text
profession
group_profession
business_category
sfera
experience
schedule
employment_type
education_level
```

Each item in `candidates` comes from Stage 1 retrieval and contains:

```text
cv_id_hash
embedding_score
embedding_rank
```

The ranking service returns a `Stage2Response`:

```text
job_id
ranked
```

Each item in `ranked` contains:

```text
cv_id_hash
rank
model_score
```

The backend remains responsible for mapping ranked CV ids into the final frontend response shape.
